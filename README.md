# Класифікатор рівнів абстракції слів (Word Abstraction Level Classifier)

Цей інструмент призначений для автоматичного аналізу та класифікації слів у базі даних (SQL-дампі) за **10 рівнями абстракції** (від Абсолюту/Буття до Одиниці/Індивіда) відповідно до онтологічних та граматичних правил.

---

## 📌 Призначення

Скрипт обробляє SQL-дамп таблиці слів (наприклад, `Word.v.10.sql`), модифікує структуру бази даних, додаючи колонку `abstraction_level`, та проставляє кожному слову відповідне значення рівня абстракції (від 1 до 10). 

Це необхідно для семантичного аналізу текстів, побудови когнітивних моделей, вимірювання логічної дистанції між поняттями («від ідеї до речі») та покращення роботи пошукових або рекомендаційних систем.

---

## ⚙️ Як це працює (Архітектура)

Для ефективної обробки великих файлів (оригінальний дамп містить ~355 тис. рядків та має розмір ~166 МБ) скрипт реалізує **двопрохідну потокову (streaming) архітектуру**:

1. **Перший прохід (Pass 1 - Scan & Heuristics)**:
   - Потоково сканує SQL-файл, вибираючи лише унікальні головні форми слів (`is_main_form = 1`).
   - Застосовує швидкі вбудовані **граматичні евристики** (імена отримують рівень 1, службові частини мови — 9 або 10, займенники — 1 або 9, числівники — 9, відомі матеріальні/абстрактні іменники тощо).
   - Визначає перелік унікальних лемм (`main_form_code`), які потребують семантичної класифікації за допомогою LLM.

2. **Проміжний крок (LLM Batch Query)**:
   - Надсилає невідомі слова пакетами (батчами по 50 слів) до обраного LLM провайдера (Gemini або OpenAI).
   - Зберігає всі результати в локальний файл кешу `classification_cache.json`. Це дозволяє у будь-який момент зупинити процес та продовжити його без повторних платних запитів.

3. **Другий прохід (Pass 2 - Rebuild SQL)**:
   - Зчитує вихідний SQL-файл рядок за рядком.
   - Модифікує блок `CREATE TABLE`, додаючи поле `abstraction_level tinyint(4)`.
   - Модифікує структуру `INSERT INTO` та значення `VALUES`.
   - Для кожної словоформи (як головної, так і відмінюваної) миттєво підтягує рівень абстракції з кешу за її `main_form_code` за час $O(1)$. Це скорочує навантаження на LLM у 10-15 разів.

## 🚀 Команди для запуску та керування

Скрипт підтримує роботу в різних режимах (через аргументи командного рядка).

### 1. Тестовий запуск на невеликому піднаборі (Heuristics Only)
Запуск класифікації на тестовому файлі (200 рядків) без використання зовнішніх LLM API:
```bash
python3 classify_words.py \
  --input base_of_word/test_words.sql \
  --output base_of_word/test_words_level.sql \
  --heuristics-only
```

### 2. Тестовий запуск на великому дампі з лімітом рядків
Сканує великий файл, але обробляє та записує лише перші 1000 слів для перевірки коректності SQL-синтаксису:
```bash
python3 classify_words.py \
  --input base_of_word/Word.v.10.sql \
  --output base_of_word/Word.v.10.level.1000.sql \
  --heuristics-only \
  --limit 1000
```

### 3. Режим повної класифікації з Gemini API (Рекомендовано)
Для роботи потрібен ключ `GEMINI_API_KEY`. Скрипт автоматично спробує взяти його зі змінних оточення або через аргумент `--api-key`:
```bash
# Встановлення ключа в оточення
export GEMINI_API_KEY="ваш_api_key_тут"

# Запуск
python3 classify_words.py \
  --input base_of_word/Word.v.10.sql \
  --output base_of_word/Word.v.10.level.sql \
  --api-type gemini \
  --model gemini-2.5-flash
```

Або передача ключа безпосередньо у команду:
```bash
python3 classify_words.py \
  --input base_of_word/Word.v.10.sql \
  --output base_of_word/Word.v.10.level.sql \
  --api-type gemini \
  --api-key "AIzaSy..."
```

### 4. Режим класифікації з OpenAI API
Використовує GPT-4o-mini для семантичного аналізу слів:
```bash
export OPENAI_API_KEY="ваш_openai_key_тут"

python3 classify_words.py \
  --input base_of_word/Word.v.10.sql \
  --output base_of_word/Word.v.10.level.sql \
  --api-type openai \
  --model gpt-4o-mini
```

### 5. Режим класифікації з динамічним LLM-ротатором (LLMRotator)
Використовує автоматичне каскадне перемикання між хмарними та локальними провайдерами (Groq, Cerebras, SambaNova, Nvidia, Cloudflare, OpenRouter, Ollama) у разі помилок лімітів (Rate Limits / 429):
```bash
python3 classify_words.py \
  --input base_of_word/Word.v.10.sql \
  --output base_of_word/Word.v.10.level.sql \
  --api-type rotator
```

Ви також можете передати назву пріоритетного (Preferred) провайдера через аргумент `--model`:
```bash
python3 classify_words.py \
  --input base_of_word/Word.v.10.sql \
  --output base_of_word/Word.v.10.level.sql \
  --api-type rotator \
  --model groq
```

## 🗂 Структура Рівнів Абстракції

Слова класифікуються за наступною шкалою:

1. **10. Абсолют (Буття)**: *всесвіт, буття, космос, є, існувати, ніщо*.
2. **9. Сутність (Концепт)**: *час, простір, кохання, ідея, число, мислення, теорія*.
3. **8. Матеріальність (Тіло)**: *матерія, вода, газ, metal, камінь, об'єкт*.
4. **7. Походження (Генезис)**: *природа, організм, тварина, рослина, виріб, файл, програма*.
5. **6. Призначення (Роль)**: *транспорт, меблі, одяг, їжа, посуд, зброя*.
6. **5. Категорія (Кластер)**: *стілець, стіл, тарілка, автомобіль, літак, будинок*.
7. **4. Рід (Гіперонім)**: *письмовий стіл, кухонний стіл, легковий автомобіль*.
8. **3. Вид (Гіпонім)**: *парта, секретер, пуховик, позашляховик*.
9. **2. Модель (Варіація)**: власні назви брендів/моделей (*Corolla, iPhone, Бекрант*).
10. **1. Одиниця (Індивід)**: власні назви унікальних істот/місць (*Київ, Сонце, Шевченко, Петро, це*).

## 🔗 Зв'язування слів між рівнями абстракції (`word_relationship.py`)

Новий модуль `word_relationship.py` призначений для встановлення ієрархічних зв'язків типу "батько-дитина" (гіперонім-гіпонім або концепт-екземпляр) між словами суміжних рівнів абстракції.

### 📌 Призначення та логіка зв'язків
Скрипт послідовно рухається згори донизу:
* **Level 10 (Абсолют)** є вершиною ієрархії (не має батьків).
* **Level 9 (Сутність)** зв'язується з **Level 10** (наприклад, поняття часу або мислення відносяться до *буття*).
* **Level 8 (Матеріальність)** зв'язується з **Level 9** (наприклад, конкретна речовина відноситься до *матерії*).
* ... і так далі до найнижчого **Level 1**.

У базі даних зв'язок реалізується через поле `parent_id`, яке вказує на первинний ключ `id` батьківського слова (тільки для головних форм `is_main_form = 1`).

---

### ⚙️ Алгоритм роботи
Програма використовує аналогічну двопрохідну потокову архітектуру для роботи з гігантськими файлами:

1. **Етап 1: Аналіз та збір лемм (Pass 1)**
   * Зчитує SQL-файл для побудови словника унікальних лемм (`main_form_code`) та розподілу їх за рівнями абстракції.
2. **Етап 2: Побудова зв'язків**
   * **Оптимізовані евристики (Python)**:
     * Для переходів **9 -> 10** та **7 -> 8** застосовуються швидкі вбудовані правила (поділ слів на категорії «космос», «існування», «істоти», «рослини», «світло», «вода» тощо та миттєве призначення батька без запитів до LLM).
     * Це дозволило заощадити понад 30,000 запитів до ШІ.
   * **Групові запити до ШІ (LLM)**:
     * Для інших рівнів (наприклад, зв'язування видів з родами чи категорій з призначеннями) слова надсилаються пакетами (батчами по 50 слів) до LLM.
     * Усі результати зберігаються в локальному кеші [relationship_cache.json](file:///Users/kostantinkrivula/Desktop/sqlbase/skill_for_word/relationship_cache.json).
3. **Етап 3: Запис результату в новий SQL (Pass 2)**
   * Створює вихідний файл (наприклад, `Word.v.10.level.relationship.sql`).
   * Модифікує таблицю `word`, додаючи колонку `parent_id` та індекс `KEY parent_id (parent_id)`.
   * Записує кожен рядок даних, підставляючи відповідний `parent_id` на основі `main_form_code` за час $O(1)$.
